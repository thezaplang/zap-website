## What You Can Build With Zap

<CardGrid stagger>
  <Card title="ARC and weak refs" icon="setting">
    Heap-owned classes use ARC, support <code>deinit()</code>, and can break cycles with <code>weak</code>, <code>alive()</code>, and <code>lock()</code>.
  </Card>

  <Card title="Classes and inheritance" icon="star">
    Zap supports heap-only classes, instance methods, visibility modifiers, single inheritance, and dynamic dispatch.
  </Card>

  <Card title="Modules and stdlib" icon="puzzle">
    Import local modules or <code>std/...</code> modules for IO, strings, process, filesystem, paths, math, memory, and conversions.
  </Card>

  <Card title="Structured low-level access" icon="error">
    Raw pointers, casts, manual allocation, and pointer arithmetic are available behind explicit <code>unsafe</code>.
  </Card>

  <Card title="Practical syntax" icon="information">
    Structs, enums, aliases, arrays, references with <code>ref</code>, ternary expressions, overloads, named args, and varargs.
  </Card>

  <Card title="Native pipeline" icon="star">
    The compiler uses an LLVM backend and can emit executables, object files, textual LLVM IR, or ZIR.
  </Card>
</CardGrid>

## Why Zap

Zap aims for systems-level control without forcing you into a garbage collector or making everything manual from day one. The language already focuses on a few concrete strengths:

- deterministic ARC for object lifetimes
- expressive data modeling with structs, enums, and classes
- a small but useful standard library
- optional `unsafe` when you need raw pointers or manual allocation
- native compilation through `zapc`
